# Frontend Trainee Spring 2026

Личный кабинет продавца с AI-ассистентом для улучшения объявлений.

## Стек

- **Frontend:** React 19 + TypeScript, Vite, MUI, TanStack Query, Zustand, Axios
- **Backend (mock):** Fastify 5 + Zod, Node.js 20+
- **Тесты:** Vitest
- **Линтер/форматтер:** ESLint + Prettier

## Запуск (локально)

### 1) Backend
```bash
cd server
npm install
npm start
```
Сервер запускается на `http://localhost:8080`.

### 2) Frontend
```bash
cd client
npm install
# Создайте .env из шаблона и при необходимости задайте VITE_API_URL
cp .env.example .env
npm run dev
```
Клиент запускается на `http://localhost:5173`.

## Запуск через Docker Compose

```bash
cd Frontend-trainee-assignment-spring-2026
docker compose up --build
```
- Backend: `http://localhost:8080`
- Frontend: `http://localhost:80`

> Если нужен другой адрес API в контейнере, передайте аргумент сборки:
> `docker compose build --build-arg VITE_API_URL=http://server:8080`

|---|---|---|

## Настройка LLM (Ollama)

```bash
# Установите Ollama: https://ollama.com/
ollama pull llama3.2
ollama serve
```

Кнопки «Придумать описание» и «Узнать рыночную цену» на странице редактирования объявления обращаются к Ollama. Без запущенного Ollama кнопки покажут toast с ошибкой — остальной функционал работает независимо.

### Улучшение качества ответов (рекомендуется)

Маленькие модели иногда смешивают языки и дают менее точные ответы по цене и описанию. Можно переключиться на более сильную модель:

```bash
ollama pull llama3.1:8b
# или
ollama pull qwen2.5:7b
```

В `client/.env`:

```env
VITE_OLLAMA_MODEL=llama3.1:8b
```

После изменения `.env` перезапустите `npm run dev`.

## Архитектура

```
src/
  api/        — HTTP-клиент, запросы к API и Ollama
  components/ — Общие UI-компоненты (AppFrame, ErrorBoundary, LoadingState, ErrorState)
  constants/  — Конфигурация категорий, полей форм, сортировки
  hooks/      — Кастомные хуки (useDebounce)
  pages/      — Страницы: AdsListPage, AdDetailsPage, AdEditPage, NotFoundPage
  store/      — Zustand: хранилище фильтров списка
  types/      — TypeScript-типы предметной области
  utils/      — Утилиты: format, revision (+ тесты)
  test/       — Конфигурация тестов
```

## Реализованный функционал

### Обязательное
- `/ads` — список с поиском (debounce 400 мс), фильтрами по категории, переключателем «только с доработками», сортировкой (по новизне / названию), пагинацией (10/стр.), переключателем вида сетка/список, placeholder-изображениями в карточках
- `/ads/:id` — карточка объявления с названием, ценой, датами публикации/редактирования, характеристиками, описанием, рядом thumbnail-placeholders; предупреждение «Требуются доработки» с перечнем незаполненных полей
- `/ads/:id/edit` — форма редактирования с динамическими полями по категории, валидацией обязательных полей, черновиком в `localStorage`, редиректом на карточку после успешного сохранения
- Состояния загрузки и ошибок на каждой странице
- `ErrorBoundary` для перехвата критических ошибок UI
 - Тёмная тема с сохранением выбора в `localStorage`

### AI-функции
- «Придумать описание» / «Улучшить описание» — генерирует текст через Ollama, позволяет применить результат в форму; есть визуальное сравнение «Было → Стало» с подсветкой добавленного/удалённого
- «Узнать рыночную цену» — возвращает рекомендации по цене (с учётом «Состояние»: новое/б/у)
- «Повторить запрос» — повторяет последний AI-запрос
- Чат с AI на странице редактирования: уточняющие вопросы по объявлению, история сообщений отображается в интерфейсе (контекст передаётся автоматически)

### Дополнительное (со звёздочкой)
- Отмена HTTP-запросов при смене страницы (через `AbortSignal`)
- Docker Compose для client + server
- Юнит-тесты: 22 теста (`utils/format`, `utils/revision`, `store/adsFilters`)

## Принятые решения и компромиссы

- **Архитектура** — упрощённая feature-flat без FSD: для масштаба задания достаточно, чище и проще для проверяющего.
- **Zustand** — выбран для хранилища фильтров: минимальный API, не требует провайдеров, легко тестируется.
- **`keepPreviousData`** — при смене фильтров старые данные остаются видны до загрузки новых, не мерцает пустой экран.
- **Черновик через `localStorage`** — debounce 400мс перед записью, чтобы не нагружать хранилище при каждом keystroke.
- **Заглушка вместо реальных изображений** — согласно ТЗ, загрузка изображений не требуется.
- **`GET /items` расширен полем `id`** — в оригинальном mock ответ не содержал `id`, добавлено для корректной навигации по карточкам.
- **Категория `auto` отображается как «Авто»** — соответствует макету Figma; ТЗ упоминает «Транспорт» как пример, но визуальный дизайн использует «Авто».
- **Сортировка по цене не реализована** — backend API не поддерживает `sortColumn=price`; сортировка по новизне и названию покрывает основной сценарий.
